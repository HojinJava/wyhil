# hax-web/jpa_to_mysql [L3] 분석

## 이해한 내용

- 목표는 JPA 기반 도메인을 MyBatis 기반 계층으로 역마이그레이션할 때 필요한 산출물과 규칙을 빠짐없이 정리하는 것이다.
- 산출물은 DTO, DAO 인터페이스, Mapper XML, Service, 필요 시 Controller까지 전체 파일 단위로 완전한 코드여야 한다.
- 핵심 제약은 JPA 흔적 제거, `Optional` 반환 유지, `@Param` 적용, XML 기반 페이징 분리, 더티 체킹을 명시적 `update(...)` 호출로 대체하는 것이다.
- API 스펙은 유지해야 하며, 패키지/임포트/네임스페이스 일관성이 중요하다.

## 현재 hax-web 기준 분석

- 현재 `hax-web`은 `pom.xml` 기준으로 `mybatis-spring-boot-starter`를 사용하고 있고 `@MapperScan("com.hnix.sd")`도 이미 설정되어 있다.
- 저장소 내 검색 결과상 `@Entity`, `JpaRepository`, `EntityManager` 같은 JPA 의존 코드는 확인되지 않았다.
- 즉, 이 이슈는 실제 변환 구현보다 "향후 JPA 도메인이 들어왔을 때 어떤 기준으로 MyBatis 계층을 만들지"에 대한 분석 문서 성격이 강하다.
- 이미 MyBatis 중심 구조인 만큼, 새 도메인 유입 시 DAO/Mapper XML 규약과 서비스의 명시적 update 흐름을 템플릿화하는 것이 현실적인 대응이다.

## 마이그레이션 시 중점 포인트

1. DTO 분리
   엔티티 상속 구조 대신 조회/수정 목적별 DTO를 명시적으로 나눠야 한다.
2. DAO 시그니처 안정성
   기존 서비스가 기대하는 반환 타입을 유지하면서 `Optional<T>`와 `@Param` 규칙을 고정해야 한다.
3. XML 쿼리 표준화
   목록 조회와 카운트 조회를 분리하고, 조인 결과는 `resultMap`으로 명확히 매핑해야 한다.
4. 서비스 로직 보정
   더티 체킹 전제 코드를 모두 찾아서 `dao.update(...)` 호출로 바꿔야 한다.
5. 컨트롤러 계약 보존
   URL, 메서드, 요청/응답 스펙은 유지하고 내부 구현만 교체해야 한다.

## 권장 작업 순서

1. 대상 도메인의 JPA 엔티티, 리포지토리, 서비스, 컨트롤러를 수집한다.
2. 엔티티를 DTO와 조회 전용 모델로 분해한다.
3. 리포지토리를 DAO 인터페이스와 Mapper XML로 치환한다.
4. 서비스에서 더티 체킹, 페이징, 연관관계 로딩 로직을 명시 호출 구조로 바꾼다.
5. 마지막으로 컨트롤러 시그니처와 통합 테스트를 기준으로 회귀 검증한다.
